# CorTeX Possible Upgrades — mitigated items & stretch goals

> These are **not open bugs.** Each is already mitigated to an acceptable state today —
> the prime directive's robustness floor is met (bounded resources, transparent failure,
> no silent loss) — and carries a documented **upgrade path** to take when a real workload,
> an owner cost-call, or an ops action calls for it. They were split out of
> [`KNOWN_ISSUES.md`](KNOWN_ISSUES.md) (the bug ledger) to signal exactly that: **stretch
> goals, not defects.** IDs are preserved, so existing code comments (e.g. `// … KNOWN_ISSUES
> W-1`) still resolve here.
>
> The severity tag reflects the **upgrade's** impact class if ever taken, not an active risk:
> **S1** robustness · **S2** work-fidelity · **S3** correctness/UX · **S4** cleanup/polish.

## Dispatcher / pipeline

| # | Sev | Mitigation today & upgrade path |
|---|---|---|
| D-13 | S4 | **Sink PULL socket lost its TCP keepalive in the phase-5a `zeromq` swap.** The async `zeromq::PullSocket` (the new sink transport, commit 9990168) exposes no TCP-keepalive knob, so `dispatcher.tcp_keepalive_idle_seconds` no longer applies to the **sink** (it still applies to the libzmq ventilator until 5b). Keepalive is **stability-only** — it keeps idle remote-worker *result* connections from being dropped by NAT/firewall idle-timeouts; correctness is unaffected (the lease reaper re-dispatches any task whose result never arrives, and a worker reconnects on its next push). So a remote worker behind NAT that sits idle past the NAT timeout may have its result connection silently dropped and need a reconnect. **Fix options:** set the keepalive via a raw socket option if `zeromq` ever exposes one; or rely on the worker's own reconnect + the reaper (current behaviour). Revisit only if remote-result NAT drops are observed in the field. |

## Worker / task handling

| # | Sev | Mitigation today & upgrade path |
|---|---|---|
| W-1 | S1 | **No per-task timeout / resource cap at the worker boundary.** `latexml-oxide` can hang, OOM, or segfault on hostile input. **Scoped (2026-06-14) — what's covered vs. not:** ① a worker that **hangs / dies** is *covered on the dispatcher side* — the reaper time-boxes dispatched tasks (`expected_at` deadline + `MAX_DISPATCH_RETRIES` → retry-or-`Fatal`, `REAP_INTERVAL_SECS` cadence, KNOWN_ISSUES D-6), so a stuck task drains and is retried/failed rather than wedging the pipeline; ② a worker's **own** memory/CPU (the external `latexml-oxide` process) can't be capped from CorTeX — that's a `pericortex`/cgroup/`ulimit` concern outside this repo. **③ Oversized result — now FIXED (2026-06-14).** The sink streamed the worker's result archive straight to `/data` tracking `total_incoming` but never bounding it — a runaway/malicious worker or a decompression bomb could write an unbounded archive and **fill `/data`** (principle #6). Now bounded by a configurable hard cap `dispatcher.max_result_bytes` (default **2 GiB**; editable on **both** consumer surfaces — the agent `GET`/`PUT /api/config` and the human **Settings** screen's "Max result archive size" field, 2026-06-16, so the safety cap is tunable without hand-editing the config file): on overflow the sink stops writing, **drains the remaining ZMQ frames frame-by-frame to keep the PULL socket frame-aligned** (a misaligned drain would cascade into every later result), removes the partial file, and finalizes the task **`Invalid`** (`result_too_large`) so it isn't leaked (`sink.rs`, `config.rs`). **Torture-tested** at real scale (`tests/dispatcher_torture_test.rs`, `CORTEX_TORTURE_BIG=1`): a 1.99 GB result accepted + written, a 3 GB result rejected/cleaned, alongside a 200k malformed-reply barrage — the sink stays frame-aligned (every concurrent real task still finalizes). **Residual:** only the out-of-repo ② (worker's own resource limits) remains; everything CorTeX *can* control at this boundary is now covered. Principles #3, #6. |
| W-4 | S3 | **No per-job timeout — a hung job thread leaks (owner: long-lived frontends must not accumulate zombies → OOM).** Audit (2026-06-14): CorTeX spawns **no subprocesses** (workers are external ZMQ processes, not children), and all threads are bounded — the dispatcher's fixed vent/sink/finalize + one metadata writer, and frontend `jobs::spawn_job` threads that **complete and are OS-reaped** (now panic-safe, so they always terminate; the shared r2d2 pool bounds their connections). The residual: a job body that **hangs** (e.g. the importer blocked forever on a stale mount) is never interrupted — its thread + pooled connection leak for the process's life; enough of them exhausts the pool / RAM. Mitigations in place: panic-safety (crashes → `failed`), and the pending/health observability surfaces a stalled job. **Observability advanced (2026-06-14):** `JobDto` now carries `seconds_since_update` — the **heartbeat age** (`db_now - updated_at`, measured against the DB clock via `jobs::db_now` so it's tz-skew-free), exposed on `GET /api/jobs[/<uuid>]` and the `/jobs` dashboard ("Idle (s)" column). For a running job it climbs while no progress is made, so a stalled body is *visible* to agents and admins without any auto-kill / threshold guess (judgment-free raw signal). Tested (`jobs_api_test`: a seeded 1h-stale running job reports a large heartbeat age). **Runtime reaper landed (2026-06-14):** `jobs::reap_stale` (called by `list_recent`, so every jobs listing / pending-check / report-refresh debounce runs it) flips any non-terminal job **whose progress heartbeat has been silent past `STALE_JOB_HEARTBEAT_TIMEOUT_SECS` (2h)** to `interrupted` — the runtime complement to the startup `interrupt_orphans`. This deliberately keys off **heartbeat staleness, not total duration**, which is what made the earlier deferral's blocker (false-killing legitimately-long reindex/refresh ops) moot: an op that keeps `step`-ing stays live regardless of how long it runs; only a *silent* (hung) body is reaped. It is also **self-correcting** — a merely-slow job that is reaped and then finishes has its status overwritten by `finish()`, so the generous threshold costs at most a transient `interrupted`. This closes the observable-zombie gap (a hung job no longer sits `running` forever, lying to every pending-check) **and** a latent deadlock (a hung `refresh_reports` job no longer blocks the debounce from ever spawning another). Tested (`jobs_api_test::stale_running_job_is_reaped_but_fresh_one_survives`). **Residual (true Rust limitation):** a body that hangs *forever* still leaks its thread + one pooled connection until process restart (Rust can't force-cancel a thread); mitigated by the pool bound + `interrupt_orphans` at restart. Bounding the operations themselves (importer blocking-I/O timeouts; `statement_timeout`/`lock_timeout` on the refresh/reindex connections) remains the deeper fix. **Threshold made operator-tunable (2026-06-15):** the hardcoded 2h is now `config.jobs.stale_timeout_seconds` (figment; scaffolded into `cortex.toml`), so a deployment running long single-statement jobs that don't heartbeat between steps can raise it without a recompile — **and now manageable on both consumer surfaces** (2026-06-15): it is exposed in `ConfigDto` (so the agent `GET /api/config` reads it and `PUT /api/config` merges it) and on the human **Settings** screen as an editable "Background jobs → Stall-reap threshold" field, so no hand-editing of the config file is needed (the "manage all configurations" goal). Pinned by `management_api_test` (contract) + `settings_test` (form + `PUT` round-trip persist). **Architecture note (why no in-app supervisor):** a brief design pass considered a background watchdog thread; it was **rejected as redundant** — `reap_stale` already runs *on every read* of the jobs surface, so every consumer self-heals the accounting, and a sweeper can't reclaim the leaked thread/RAM anyway. The framework can't help here either: these jobs are **detached** `std::thread::spawn` workers (the request already returned `202`), outside Rocket's request lifecycle; and even under tokio a **blocking** body (diesel/libpq, `REINDEX`) is uncancellable (`timeout`/`spawn_blocking` can't abort mid-call). The **only** true fix for force-killing a hung blocking job is **subprocess isolation** (a SIGKILL-able child) — a deliberate architecture trade, not built (not worth it for rare admin-job hangs). So this stays 🟡 by design: accounting is correct + tunable; the un-killable-thread leak is an accepted, documented in-process limit. |

| I-2 | S4 | **`activate`/`extend` load a corpus's entire import-entry list into RAM.** `services_aggregate.rs` (`Backend::activate`/`extend`, reached from the web `/services/<s>/activate` + the agent `POST /api/corpora/<c>/services/<s>` and `…/extend`) does `tasks.filter(import service).filter(corpus).select(entry).load()` — every imported document's `entry` path (up to ~1.5M strings ≈ ~120 MB for arxmliv) materialized in one `Vec`, then re-creates one TODO conversion task per entry inside a single transaction. **Same unbounded-work-list class as the (now-fixed) E-3**, but on the corpus-activation *write* path rather than the export *read* path. **Mitigated:** it is a **one-time, admin-gated** operation, run as a **background job** by the web/agent (the request returns immediately; the RAM lives in the job thread, bounded by a single corpus's size), so it neither blocks a request nor scales with traffic — and the box already handles corpus-scale data (the importer walk, the dispatcher fleet). An existing `// TODO … Currently we try to reinsert all imported tasks, which is wasteful` (services_aggregate.rs:~118) already flags the re-insert inefficiency. **Upgrade path:** keyset-paginate the entry scan (`entry > $cursor ORDER BY entry LIMIT N`, the pattern E-3 now uses) and create tasks in batches, and/or make `extend` insert only genuinely-**new** documents (the TODO's point) instead of reinserting all. Touches the **destructive** (re)activation transaction (the current single-transaction atomicity is a feature), so it needs care; deferred until a memory-constrained box or a ≫1.5M-doc corpus makes it bite. |

## Performance

| # | Sev | Mitigation today & upgrade path |
|---|---|---|
| P-2 | S3 | **`?all=true` report is a multi-second live aggregation (owner cost-call; found via latency test-drive 2026-06-16).** The "all messages" report variant (`GET /corpus/<c>/<s>/<severity>?all=true` and its agent twin) bypasses the `report_summary` rollup and runs a **live** `tasks LEFT OUTER JOIN log_<severity> … GROUP BY (category, task_id)` over **every completed task** of the corpus (`task_report_live`, the `all_messages` branch), then re-aggregates by category. On `cortex_load`/arxmliv this is **~8.8 s (warning) / ~5.7 s (error)**, scaling with corpus size (a 91-doc corpus is 0.19 s) — it aggregates tens of millions of message rows (the top warning category alone is 33.7M messages). The plain per-severity report is rollup-backed and **~0.01–0.09 s**; only the `all=true` toggle is slow. **Robustness caveat:** each such request holds one pooled connection for the full ~9 s, so a handful of concurrent `?all=true` hits (a curious user, or a scripted `/api` crawl) can saturate the frontend pool (`pool_size`, default 32) and make *other* requests `503` on `pool.get()` — a real availability risk from an opt-in path (cf. the resource-exhaustion class, principle #6). **No cheap query win:** the dominant cost is aggregating the real message rows, not the `LEFT` join's empty bucket, so an `INNER` join wouldn't materially help; the only real fix is **pre-aggregation** — extending `report_summary` with the all-messages (cross-severity) grains, which the owner flagged as a **cost call** (≈ doubling the matview; the freshness-oracle plumbing is already in place — see the memory note). Deliberately **not** mitigated unilaterally (a `statement_timeout` or a fall-back-to-rollup redirect would change the feature's behaviour/UX — owner's call). Until then: prefer the per-severity views; treat `?all=true` as an expensive admin query, not an agent hot path. **Observability landed (2026-06-16, ungated):** `serve_report` now emits a structured `tracing::warn` for any live (non-rollup) report that holds its pooled connection past `SLOW_REPORT_WARN_MS` (2 s) — `slow live report held a pooled connection… corpus=… severity=… all_messages=… duration_ms=…` — so an operator can finally measure *how often* the slow `all=true` toggle is actually hit and *how slow* it runs in production, which is the evidence the cost call needs (no behaviour change; pure logging). This required giving the frontend its **own** `tracing` subscriber (`init_tracing()` in `bin/frontend.rs`): Rocket 0.5 logs via `tracing` but its default subscriber ignores `RUST_LOG` and drops app-target events, so every frontend `tracing` event was previously lost (legacy code reached for `println!`). The subscriber defaults to `info,rocket=warn` so the dashboard's `/admin/status.json` poll doesn't flood the log; `RUST_LOG` overrides. **Behaviour-preserving mitigation option for the owner (3rd path, not in the list above):** a small **bounded-concurrency limiter** on the live path — acquire one of *N* (e.g. 4) permits from a `tokio::sync::Semaphore` (held in managed state) *before* `pool.get()` in the `all=true`/live branch, release after. The expensive aggregation is then capped at *N* concurrent (≤ *N* pooled connections), leaving the rest of the pool for cheap requests, so a burst can no longer `503` everyone. Unlike a `statement_timeout`/redirect it changes **no** result and rejects **nothing** — the *(N+1)*th expensive request simply queues briefly (added latency on an already-slow opt-in path), i.e. pure blast-radius isolation (principle #6). Cheaper than the matview and reversible; offered for the owner's decision alongside the pre-aggregation cost call. **DECISION 2026-06-16 — owner chose the bounded-concurrency semaphore** (not the matview pre-aggregation): protect availability cheaply and reversibly without changing any result, scheduled after the Arm 3 data-integrity work. **LANDED 2026-06-16:** `LiveReportLimiter` (`frontend::concerns` — a `tokio::sync::Semaphore` of `MAX_CONCURRENT_LIVE_REPORTS` = 4) is managed in `mount_api_with`; the two human `?all=true` routes (`severity_service_report_all`/`category_service_report_all`) `acquire().await` a permit **before** `pool.get()` (a waiting request holds no connection and no worker — the wait is async) and hold it across the live aggregation; cheap rollup/paged requests skip the permit. The agent `/api/reports/*` endpoints are **rollup-only** (never live), so no agent-side gating is needed — the live path is human-routes-only. Unit-tested (cap + RAII release; `caps_then_releases_permits`) + `reports_api_test` exercises the gated `?all=true` path. **Now 🟡 not 🟢:** the ~9 s *latency* itself is unchanged (the matview pre-aggregation was deliberately not built); only the pool-saturation/`503` blast radius is closed (≤ 4 live aggregations, ≤ 4 pinned connections, at any time). |

## Deployment / edge exposure

| # | Sev | Mitigation today & upgrade path |

| X-3 | S4 | **Real client IP is not threaded end-to-end through the edge — a latent prerequisite, not a live defect.** The walled HTML path is 3 hops (Caddy→Anubis→Caddy router→cortex); `X-Real-Ip` is set at the public Caddy but Anubis/the router may rewrite it, so a real client IP would arrive degraded on report pages (the single-hop `/api` exempt path keeps it). **Finding (2026-06-16, why this is an upgrade not a bug):** cortex currently captures/consumes **no** client IP anywhere — no Rocket `ip_header` config, no `client_ip`/`remote()` use, no IP column in the operational journal — so *nothing on any screen or in any log is actually wrong today*; the degraded value has no consumer. **Upgrade path (do this first, only if/when a per-IP feature lands** — abuse rate-limiting, audit-trail IP enrichment): confirm Anubis forwards `X-Real-Ip` to its upstream (or thread one trusted forwarded header end-to-end), set Rocket `ip_header` to it, then add the consumer. Until such a feature exists there is nothing to fix in cortex. |
