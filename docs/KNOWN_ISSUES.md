# CorTeX Known Issues — resilience & correctness ledger

> Running list of known problems, weighted toward **robustness/fault-tolerance** gaps (see
> [`DESIGN_PRINCIPLES.md`](DESIGN_PRINCIPLES.md)). Owner's direction: **record every known problem as
> we find it; we go back and solve them all at the end.** Do not silently fix-and-forget, and do not
> silently leave a discovered gap unrecorded.
>
> Status legend: 🔴 open · 🟡 partially mitigated · 🟢 resolved (kept for history).
> Severity: **S1** can crash/corrupt/destabilise the system · **S2** drops or hides work ·
> **S3** correctness/UX papercut · **S4** cleanup/polish.

## Dispatcher / pipeline

| # | Sev | Status | Issue |
|---|---|---|---|
| D-1 | S1 | 🟢 | **Per-event connection/thread storm in `worker_metadata` — fixed.** `record_*` formerly opened a fresh `PgConnection` *and* spawned a detached thread per ZMQ event; under load this exhausted `max_connections` + ephemeral ports and **crashed the process** (proven, `RESOURCE_RATIONALIZATION.md`). #4 pooled the connection; this arm replaces the per-event spawn with a **single background writer** (`start_metadata_writer`) fed by a bounded, non-blocking `sync_channel` — **O(1) threads** regardless of dispatch rate, ≤1 metadata connection in use at a time, sends never block the dispatch hot loop, and a saturated queue drops (rather than OOMs or stalls). The ventilator/sink hold a cloneable `WorkerMetadataSender`; the writer exits when all senders drop. |
| D-2 | S2 | 🟢 | **Metadata read-before-insert race — fixed.** `record_received` did find-then-update and silently dropped the count when the sink's write outran the ventilator's insert; with no uniqueness, concurrent inserts could also duplicate rows. Now both writers **upsert** (`INSERT … ON CONFLICT (name, service_id) DO UPDATE`) via synchronous `upsert_dispatched`/`upsert_received` helpers — order-independent, no dropped counts, one row per worker (migration `20260613160000` adds `UNIQUE(name, service_id)` after a one-time dedupe). Failures now `eprintln!` instead of `.unwrap_or(0)` swallowing them. Unit-tested (out-of-order + accumulation). The thread-per-event spawn is still unbounded — see D-1. |
| D-3 | S1 | 🟢 | **RESOLVED — full panic-site audit (2026-06-14; see `docs/DISPATCHER_RATIONALIZATION.md`).** Every `src/dispatcher/` panic site classified: the mutex-poison `.expect("…Mutex lock…")`s are the **deliberate** abort-on-poison (CLAUDE.md design); the two queue `panic!`s are **deliberate** `*_QUEUE_HARD_LIMIT` fail-fast backstops; the `server.rs:309/314` `panic!`s are `#[cfg(test)]`; **`connection_at` is retry-hardened** (5 attempts + backoff, panics only when the DB is truly down); and every worker-thread death is **supervised** — the manager `join()`s the ventilator and polls `sink/finalize.is_finished()` every 1s → `Err(ETERM)` → process abort → external restart (no silent dead thread; the comment in `manager.rs` is explicit). Jobs additionally `catch_unwind`. **No accidental crash/silent-death gaps remain on the dispatch path.** What's left is *throughput rationalization* (lock-free/fan-out, owner directive 2026-06-14), which is a separate concern, not a resilience bug — see the rationalization doc. *(Original note: `.expect()`/`.unwrap()`/`panic!` on dispatch paths.)* |
| D-4 | S2 | 🟡 | **Ventilator empty-message fragility** (pre-existing, code-commented). A rare "3 adjacent empty messages" mode permanently shuffles ROUTER state; the manager works around it by **restarting the ventilator thread in a loop**. The ROUTER-framing root cause is still open (hard to reproduce). **One real correctness interaction of the restart band-aid is now FIXED (2026-06-14):** `Ventilator::start` calls `clear_limbo_tasks` *per (re)start* (`ventilator.rs:59`), which bluntly reset **all** `status>0` (Queued) tasks → TODO. At process start that's correct crash recovery, but on a *mid-operation restart* the sink is still processing **in-flight** tasks (held in `progress_queue`, `status=Queued`) — resetting those re-leased them while their original results were still pending (a **double-dispatch**: wasted compute, the duplicate's result later discarded as "no such task"). Fixed by excluding the live `progress_queue` ids: new `Backend::clear_limbo_tasks_except(&in_flight)` (`tasks_aggregate::clear_limbo_tasks_except`, `id <> ALL(in_flight)`), with the ventilator passing its `progress_queue` snapshot; an empty slice is the old blunt reset, so process-start is unchanged. Tested (`backend_test::clear_limbo_except_preserves_in_flight_tasks`; `task_lifecycle` + `echo_roundtrip` confirm process-start behaviour unchanged). **Residual:** a microsecond race remains (a task popped from `progress_queue` by the sink *exactly* during the restart's snapshot) — negligible and non-corrupting (mark_done is idempotent); a fully race-free version needs transactional coordination (owner design). The ROUTER-framing root cause (avoiding the restart entirely) remains the open S2 work. |
| D-5 | S2 | 🔴 | **`job_limit` lockstep termination hangs — root cause precisely diagnosed (2026-06-14).** With a finite `job_limit`, the pipeline can **hang on shutdown** (hit while writing `bench_pipeline.rs`). **Why (the three threads count `job_limit` in three *incompatible units*):** ① the **ventilator** (`source_job_count`, `ventilator.rs:94/215`) increments per **request** and terminates at `>= limit` — but that count includes **every mock-reply** (unknown-service, backpressure, *and* momentary-empty-queue, taskid `0`), so it terminates after `limit` *requests* having dispatched *fewer than `limit`* real tasks; ② the **sink** (`sink_job_count`, `sink.rs:69`) counts **results received** (= completed tasks); ③ the **finalize** thread (`jobs_count`, `finalize.rs:44/65`) increments **per non-empty drain cycle**, where `server::mark_done_arc` `.drain(..)`s the *entire* `done_queue` each call — so one increment = a whole variable-size batch, not one task. With these mismatched units the threads can never agree on "done": the ventilator stops early (mock-replies inflate its count) while finalize blocks waiting for `limit` drain-cycles / the sink waits for results that were never dispatched → deadlock. **Fix is a coordination redesign, not a patch:** one shared completion condition — e.g. a shared atomic *dispatched-real-task* counter, finalize/sink terminating when *finalized == dispatched* once the ventilator signals "source exhausted", and an explicit "DB has no more TODO tasks" drain (today the empty-queue case just mock-replies forever). Counting must be unit-consistent (finalize should `+= reports.len()`, not `+= 1`). Touches all three threads + `bench_pipeline` expectations, so it needs owner-reviewed design (OPEN_QUESTIONS). The **perpetual production dispatcher runs with `job_limit = None`**, so this is benchmark/bounded-run-only — not a production-path hang. |
| D-6 | S2 | 🟢 | **Dispatch backpressure — implemented (2026-06-13).** Previously the metadata fan-out was bounded (D-1) but nothing bounded the **in-flight task set**: the ventilator fire-and-forgot, and `progress_queue` could only grow until the hard `PROGRESS_QUEUE_HARD_LIMIT` (10k) **panic** — i.e. crash, not degrade. Now the ventilator applies **backpressure** (principle #4): when in-flight ≥ `DispatcherConfig::max_in_flight` (default 5000, well under the 10k hard bound — an invariant asserted in `server::tests`), it stops leasing and mock-replies so workers back off and retry; the set then drains as the sink receives results, holding steady instead of growing to the panic. The two magic 10k bounds are now the named `PROGRESS_QUEUE_HARD_LIMIT` / `DONE_QUEUE_HARD_LIMIT` constants (the hard backstop behind the soft backpressure). Unit-tested (`in_flight_saturated` boundary, the below-hard-limit invariant, `progress_queue_len` dispatch/drain); `echo_roundtrip` (full dispatcher) green. **Reaping residual now CLOSED (2026-06-13):** timeout reaping was decoupled from the refetch path onto a fixed `REAP_INTERVAL_SECS` (60s) cadence at the top of the dispatch loop, so the in-flight set drains even while saturated (when refetch never runs). The reap logic was extracted into pure, unit-tested helpers (`classify_expired` → retry-or-`Fatal`, `reap_expired_into`); in doing so a **latent cross-service requeue bug** was fixed — expired tasks were re-queued into the *requesting* worker's service queue, now the dispatch queues are keyed by `service_id` and each reaped task returns to **its own** service (so a service-B task can't be dispatched to a service-A worker). |
| D-7 | S3 | 🔴 | **Single sink thread does blocking result-archive writes** to slow `/data` QLC RAID6 — serialises I/O at high task rates. **Now folded into the dispatcher rationalization** (owner directive 2026-06-14): phase 2 splits the sink into a receive loop + a fanned-out writer pool fed by a bounded channel, so receiving is no longer hostage to disk latency. Plan: `docs/DISPATCHER_RATIONALIZATION.md`. (Was: Arm 14 #1/#2 NVMe staging + write thread-pool.) |
| D-8 | S3 | 🟢 | **`mark_done` write-amplification — deletes, status updates *and* inserts now all batched.** It used to issue **five deletes + one status UPDATE per task** plus **one INSERT per message** in the finalize loop (≈ `7·N + M` statements). Now it (a) collects the batch's task ids once and clears the prior logs with **one `task_id = ANY(...)` delete per table** (5 statements regardless of batch size); (b) groups the finalized ids by their **distinct target status** and issues **one `UPDATE … WHERE id = ANY(...)` per status value** (≤~6 — the fixed `NoProblem/Warning/Error/Fatal/Invalid` set — over disjoint id sets); and (c) partitions the new messages by severity and issues **one batched `INSERT … VALUES (…), (…)` per non-empty `log_*` table** (≤5). So a finalize batch of N tasks / M messages went from `≈7·N + M` round-trips to **≤16, independent of N and M** — equivalent because a finalize batch holds distinct task ids. Regression-tested (`backend_test`: a re-finalize with no messages must delete the prior logs; `mark_done_routes_messages_to_severity_tables` finalizes a mixed-severity, mixed-*status* batch and asserts each message lands in its own table **and** each task receives its own status — covering the by-status grouping). **Remaining (separate, not blocking):** the reinsert is still unconditional (no diff/upsert to skip genuinely-unchanged logs) — a smaller optional win tracked under Arm 14 #3. |
| D-9 | S2 | 🟢 | **RESOLVED (2026-06-14) — unsupervised sink/finalize threads → silent stall.** The manager (`dispatcher/manager.rs`) restart-loops only the **ventilator**; the **sink** and **finalize** threads are spawned **once**. In perpetual mode (`job_limit = None`, i.e. production) the supervision loop spins on the ventilator forever and **never reaches the sink/finalize joins**, so if either died (a panic on a DB runaway, or the sink's `return Err` on an unknown service id), the dispatcher kept running with a **dead result pipeline**: results pile up unprocessed, the in-flight set saturates, the ventilator mock-replies forever, and nothing aborts — a *silent stall* that contradicts the intended fail-fast ("process abort → external restart"). Fix: the supervision loop now polls `sink_thread.is_finished()` / `finalize_thread.is_finished()` each iteration (perpetual mode only — `job_limit` mode `break`s first, so a *cleanly finished* thread is never mistaken for a death) and **aborts with `ETERM`** if either has terminated, so the external supervisor restarts the whole dispatcher; on restart `clear_limbo_tasks` returns the leased-but-unfinished tasks to `TODO` (no work lost — crash-consistent). Happy path verified by `echo_roundtrip` (full dispatcher, `job_limit = Some(1)`). **Related:** the sink's `return Err`/`panic` triggers (e.g. `get_service` miss) are now *supervised* (→ restart) rather than silently fatal; root-causing/​hardening those individual sink panics is the remaining D-7-area work. |

## Worker / task handling

| # | Sev | Status | Issue |

| W-1 | S1 | 🔴 | **No per-task timeout / resource cap at the worker boundary.** `latexml-oxide` can hang, OOM, or segfault on hostile input. **Scoped (2026-06-14) — what's covered vs. not:** ① a worker that **hangs / dies** is *covered on the dispatcher side* — the reaper time-boxes dispatched tasks (`expected_at` deadline + `MAX_DISPATCH_RETRIES` → retry-or-`Fatal`, `REAP_INTERVAL_SECS` cadence, KNOWN_ISSUES D-6), so a stuck task drains and is retried/failed rather than wedging the pipeline; ② a worker's **own** memory/CPU (the external `latexml-oxide` process) can't be capped from CorTeX — that's a `pericortex`/cgroup/`ulimit` concern outside this repo. **The concrete CorTeX-side residual is an *oversized result* with no cap:** the sink streams the worker's result archive straight to `/data` (`sink.rs:121-136`, `while sink.recv … file.write`), tracking `total_incoming` but **never bounding it** — a runaway/malicious worker or a decompression bomb can write an unbounded archive and **fill `/data`** (principle #6, unbounded per-event acquisition). The size is already *logged* (`sink.rs:176`), just not capped. **Fix design (owner-reviewed — OPEN_QUESTIONS #11):** a configurable `dispatcher.max_result_size_bytes` cap; on overflow, stop writing, **drain the remaining ZMQ frames to keep the PULL socket frame-aligned** (a misaligned drain cascades into every later result — the same fragility class as D-4), clean up the partial file, and finalize the task `Fatal` (oversized) so it isn't leaked. Needs a cap value (safety backstop, e.g. multi-GB — clearly beyond any legitimate conversion result) and the reject-vs-truncate behaviour decided. Principles #3, #6. |
| W-2 | S3 | 🟢 | **RESOLVED (2026-06-14).** A non-UTF-8 worker log used to be discarded *wholesale* — `generate_report` replaced the entire log with a synthetic `Fatal:cortex:unicode_parse_error` + `Status:conversion:3`, throwing away every real message and force-marking the task **Fatal** over a single stray byte. Unit-tested (`helpers::log_decode_tests`, DB-free): valid UTF-8 passes through untouched; a `0xFF`-containing log keeps its real status + messages and gains the warning instead of collapsing to one fatal. |
| W-3 | S4 | 🟢 | **RESOLVED (2026-06-14) for the in-repo worker.** `InitWorker::start` *unconditionally* generated a random 19-letter ZMQ identity and set it on the socket, **ignoring the configured `identity` field** — so metadata keys were not operator-controlled, and (worse) a fresh random name each restart fragmented a worker's `worker_metadata` row across process restarts. Unit-tested (configured → verbatim; empty → 19 lowercase letters). The operator owns uniqueness (a shared DEALER identity would break ROUTER addressing). **Out of scope:** the external `pericortex` workers self-assign identities in that crate — not governed here. |
| W-4 | S3 | 🟡 | **No per-job timeout — a hung job thread leaks (owner: long-lived frontends must not accumulate zombies → OOM).** Audit (2026-06-14): CorTeX spawns **no subprocesses** (workers are external ZMQ processes, not children), and all threads are bounded — the dispatcher's fixed vent/sink/finalize + one metadata writer, and frontend `jobs::spawn_job` threads that **complete and are OS-reaped** (now panic-safe, so they always terminate; the shared r2d2 pool bounds their connections). The residual: a job body that **hangs** (e.g. the importer blocked forever on a stale mount) is never interrupted — its thread + pooled connection leak for the process's life; enough of them exhausts the pool / RAM. Mitigations in place: panic-safety (crashes → `failed`), and the pending/health observability surfaces a stalled job. **Observability advanced (2026-06-14):** `JobDto` now carries `seconds_since_update` — the **heartbeat age** (`db_now - updated_at`, measured against the DB clock via `jobs::db_now` so it's tz-skew-free), exposed on `GET /api/jobs[/<uuid>]` and the `/jobs` dashboard ("Idle (s)" column). For a running job it climbs while no progress is made, so a stalled body is *visible* to agents and admins without any auto-kill / threshold guess (judgment-free raw signal). Tested (`jobs_api_test`: a seeded 1h-stale running job reports a large heartbeat age). **Still deferred (owner decision — see OPEN_QUESTIONS):** auto-interrupting a stalled job (a watchdog flipping it to `failed`/`stalled` past a deadline) and bounding the underlying operations (importer blocking I/O timeouts; a `lock_timeout`/`statement_timeout` on the refresh/reindex connections) — both need a tuning threshold the owner should set, since legitimately-long reindex/refresh ops must not be false-killed (Rust can't force-cancel a thread, so bound the *operations*, not the thread). |

| L-1 | S2 | 🟡 | **At-exit SIGSEGV in DB-pool test processes — root cause found, being eliminated.** Integration binaries that build a Rocket `Client` over an r2d2/libpq pool SIGSEGV **during process teardown, after all assertions pass** (~4/5 for `runs_test` at baseline). **Root cause (2026-06-13):** at exit the C **`atexit` handlers** (libpq/OpenSSL global cleanup) run *concurrently with the still-live Tokio runtime + r2d2 reaper threads*, which are mid-libpq — a use-after-free → SIGSEGV.
