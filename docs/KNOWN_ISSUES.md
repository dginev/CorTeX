## Dispatcher / pipeline

| # | Sev | Status | Issue |
|---|---|---|---|
| D-1 | S1 | рџџў | **Per-event connection/thread storm in `worker_metadata` вЂ” fixed.** `record_*` formerly opened a fresh `PgConnection` *and* spawned a detached thread per ZMQ event; under load this exhausted `max_connections` + ephemeral ports and **crashed the process** (proven, `RESOURCE_RATIONALIZATION.md`). #4 pooled the connection; this arm replaces the per-event spawn with a **single background writer** (`start_metadata_writer`) fed by a bounded, non-blocking `sync_channel` вЂ” **O(1) threads** regardless of dispatch rate, в‰¤1 metadata connection in use at a time, sends never block the dispatch hot loop, and a saturated queue drops (rather than OOMs or stalls). The ventilator/sink hold a cloneable `WorkerMetadataSender`; the writer exits when all senders drop. |
| D-2 | S2 | рџџў | **Metadata read-before-insert race вЂ” fixed.** `record_received` did find-then-update and silently dropped the count when the sink's write outran the ventilator's insert; with no uniqueness, concurrent inserts could also duplicate rows. Now both writers **upsert** (`INSERT вЂ¦ ON CONFLICT (name, service_id) DO UPDATE`) via synchronous `upsert_dispatched`/`upsert_received` helpers вЂ” order-independent, no dropped counts, one row per worker (migration `20260613160000` adds `UNIQUE(name, service_id)` after a one-time dedupe). Failures now `eprintln!` instead of `.unwrap_or(0)` swallowing them. Unit-tested (out-of-order + accumulation). The thread-per-event spawn is still unbounded вЂ” see D-1. |
| D-3 | S1 | рџ”ґ | **`.expect()`/`.unwrap()`/`panic!` on dispatch paths.** `connection_at` does `establish(...).expect(...)`; the ventilator does `.bind().unwrap()`; sink/finalize/manager `panic!` on thread errors. Any of these turns a transient fault into a crash or a silently-dead thread. Violates principle #2. Fix: convert to `Result` + log + continue; reserve panics for true invariants. (The `establish` message **bug is fixed** вЂ” it now interpolates the address *and* the underlying error: `panic!("Error connecting to {address}: {e}")`. The remaining panics are still here; the boot-time ones (dispatcher/ventilator bind) are defensible fail-fast, the per-request/job `connection_at` ones are the real gap. **Mitigation landed:** `jobs::spawn_job` now `catch_unwind`s the body, so a panicking job вЂ” incl. a `connection_at` panic when the DB blips вЂ” becomes a terminal `failed` rather than a zombie stuck `running` forever.) |
| D-4 | S2 | рџ”ґ | **Ventilator empty-message fragility** (pre-existing, code-commented). A rare "3 adjacent empty messages" mode permanently shuffles ROUTER state; the manager works around it by **restarting the ventilator thread in a loop**. Needs a root-cause fix, not a restart band-aid. |
| D-5 | S2 | рџ”ґ | **`job_limit` lockstep termination hangs.** With a finite `job_limit`, mock-replies (taskid 0 when the queue momentarily empties) desync the ventilator/sink/finalize counters, so the pipeline can **hang on shutdown** (hit while writing `bench_pipeline.rs`). Clean, deterministic drain/shutdown is missing. |
| D-6 | S2 | рџџў | **Dispatch backpressure вЂ” implemented (2026-06-13).** Previously the metadata fan-out was bounded (D-1) but nothing bounded the **in-flight task set**: the ventilator fire-and-forgot, and `progress_queue` could only grow until the hard `PROGRESS_QUEUE_HARD_LIMIT` (10k) **panic** вЂ” i.e. crash, not degrade. Now the ventilator applies **backpressure** (principle #4): when in-flight в‰Ґ `DispatcherConfig::max_in_flight` (default 5000, well under the 10k hard bound вЂ” an invariant asserted in `server::tests`), it stops leasing and mock-replies so workers back off and retry; the set then drains as the sink receives results, holding steady instead of growing to the panic. The two magic 10k bounds are now the named `PROGRESS_QUEUE_HARD_LIMIT` / `DONE_QUEUE_HARD_LIMIT` constants (the hard backstop behind the soft backpressure). Unit-tested (`in_flight_saturated` boundary, the below-hard-limit invariant, `progress_queue_len` dispatch/drain); `echo_roundtrip` (full dispatcher) green. **Reaping residual now CLOSED (2026-06-13):** timeout reaping was decoupled from the refetch path onto a fixed `REAP_INTERVAL_SECS` (60s) cadence at the top of the dispatch loop, so the in-flight set drains even while saturated (when refetch never runs). The reap logic was extracted into pure, unit-tested helpers (`classify_expired` в†’ retry-or-`Fatal`, `reap_expired_into`); in doing so a **latent cross-service requeue bug** was fixed вЂ” expired tasks were re-queued into the *requesting* worker's service queue, now the dispatch queues are keyed by `service_id` and each reaped task returns to **its own** service (so a service-B task can't be dispatched to a service-A worker). |
| D-7 | S3 | рџ”ґ | **Single sink thread does blocking result-archive writes** to slow `/data` QLC RAID6 вЂ” serialises I/O at high task rates. Tracked as Arm 14 #1/#2 (NVMe staging + optional write thread-pool). |
| D-8 | S3 | рџ”ґ | **`mark_done` blind delete+reinsert** of all five `log_*` tables per task on every finalize, even when logs are unchanged вЂ” write amplification on the hot path. Tracked as Arm 14 #3 (diff/upsert). |

## Worker / task handling

| # | Sev | Status | Issue |
|---|---|---|---|
| W-1 | S1 | рџ”ґ | **No per-task timeout / resource cap at the worker boundary.** `latexml-oxide` can hang, OOM, or segfault on hostile input; nothing here time-boxes or memory-caps a task or guards against decompression bombs / oversized results. Retry budget exists in the ventilator but task-level isolation does not. Principles #3, #6. |
| W-2 | S3 | рџ”ґ | **Non-UTF-8 / malformed worker logs** are handled ad hoc in `generate_report` (defaults to Fatal). Needs a single, well-tested tolerant log parser as the contract with unpredictable workers. |
| W-3 | S4 | рџџЎ | **Worker identity is self-assigned** (`hostname:service:pid`-style), overriding the configured `identity` field вЂ” fine today, but means metadata keys are not operator-controlled. |
| W-4 | S3 | рџџЎ | **No per-job timeout вЂ” a hung job thread leaks (owner: long-lived frontends must not accumulate zombies в†’ OOM).** Audit (2026-06-14): CorTeX spawns **no subprocesses** (workers are external ZMQ processes, not children), and all threads are bounded вЂ” the dispatcher's fixed vent/sink/finalize + one metadata writer, and frontend `jobs::spawn_job` threads that **complete and are OS-reaped** (now panic-safe, so they always terminate; the shared r2d2 pool bounds their connections). The residual: a job body that **hangs** (e.g. the importer blocked forever on a stale mount) is never interrupted вЂ” its thread + pooled connection leak for the process's life; enough of them exhausts the pool / RAM. Mitigations in place: panic-safety (crashes в†’ `failed`), and the pending/health observability surfaces a stalled job. **Fix:** a job watchdog (deadline в†’ mark `failed`/`stalled`) and timeouts on the importer's blocking I/O (Rust can't force-cancel a thread, so bound the *operations*, not the thread). |

## Process lifecycle / shutdown

| # | Sev | Status | Issue |
|---|---|---|---|
| L-1 | S2 | рџџЎ | **At-exit SIGSEGV in DB-pool test processes вЂ” root cause found, being eliminated.** Integration binaries that build a Rocket `Client` over an r2d2/libpq pool SIGSEGV **during process teardown, after all assertions pass** (~4/5 for `runs_test` at baseline). **Root cause (2026-06-13):** at exit the C **`atexit` handlers** (libpq/OpenSSL global cleanup) run *concurrently with the still-live Tokio runtime + r2d2 reaper threads*, which are mid-libpq вЂ” a use-after-free в†’ SIGSEGV. Proven empirically: leaking the `Client` (threads stay live) в†’ 8/8 crash; `std::process::exit(0)` (runs atexit) в†’ still ~6/10; **`libc::_exit(0)`** (skips atexit; the OS reclaims the live threads/connections cleanly) в†’ **0/12**. **Fix (being rolled out):** the affected binaries use a **custom harness** (`harness = false`) вЂ” `main` owns the `Client`, runs the cases, and ends with `unsafe { libc::_exit(0) }` *while the Client is still alive*; a panic still aborts non-zero so real failures fail CI. `runs_test` converted (0/12). Once all are converted, `scripts/ci_test.sh`'s SIGSEGV tolerance is removed (eliminated, not tolerated). NB: this is a *test-process* exit concern; production frontends are long-lived and never hit the atexit-vs-threads race. |
