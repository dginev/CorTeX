| D-4 | S2 | 🟢 | **Ventilator request-framing fragility — RESOLVED (2026-06-14).** A rare "3 adjacent empty messages" mode permanently shuffled ROUTER state; the manager worked around it by **restarting the ventilator thread in a loop**. **The ROUTER-framing root cause is now fixed** (`ventilator.rs`): the old code read the second frame *unconditionally*, so a truncated `[identity]`-only / empty / over-long request made it read the *next* request's identity as this one's service — desyncing every later request (the "permanent shuffle") — and bailed the whole ventilator on the both-empty case. | D-5 | S2 | 🔴 | **`job_limit` lockstep termination hangs — root cause precisely diagnosed (2026-06-14).** With a finite `job_limit`, the pipeline can **hang on shutdown** (hit while writing `bench_pipeline.rs`). **Why (the three threads count `job_limit` in three *incompatible units*):** ① the **ventilator** (`source_job_count`, `ventilator.rs:94/215`) increments per **request** and terminates at `>= limit` — but that count includes **every mock-reply** (unknown-service, backpressure, *and* momentary-empty-queue, taskid `0`), so it terminates after `limit` *requests* having dispatched *fewer than `limit`* real tasks; ② the **sink** (`sink_job_count`, `sink.rs:69`) counts **results received** (= completed tasks); ③ the **finalize** thread (`jobs_count`, `finalize.rs:44/65`) increments **per non-empty drain cycle**, where `server::mark_done_arc` `.drain(..)`s the *entire* `done_queue` each call — so one increment = a whole variable-size batch, not one task. With these mismatched units the threads can never agree on "done": the ventilator stops early (mock-replies inflate its count) while finalize blocks waiting for `limit` drain-cycles / the sink waits for results that were never dispatched → deadlock. **Fix is a coordination redesign, not a patch:** one shared completion condition — e.g. a shared atomic *dispatched-real-task* counter, finalize/sink terminating when *finalized == dispatched* once the ventilator signals "source exhausted", and an explicit "DB has no more TODO tasks" drain (today the empty-queue case just mock-replies forever). Counting must be unit-consistent (finalize should `+= reports.len()`, not `+= 1`). Touches all three threads + `bench_pipeline` expectations, so it needs owner-reviewed design (OPEN_QUESTIONS). The **perpetual production dispatcher runs with `job_limit = None`**, so this is benchmark/bounded-run-only — not a production-path hang. |

| D-8 | S3 | 🟢 | **`mark_done` write-amplification — deletes, status updates *and* inserts now all batched.** It used to issue **five deletes + one status UPDATE per task** plus **one INSERT per message** in the finalize loop (≈ `7·N + M` statements). Now it (a) collects the batch's task ids once and clears the prior logs with **one `task_id = ANY(...)` delete per table** (5 statements regardless of batch size); (b) groups the finalized ids by their **distinct target status** and issues **one `UPDATE … WHERE id = ANY(...)` per status value** (≤~6 — the fixed `NoProblem/Warning/Error/Fatal/Invalid` set — over disjoint id sets); and (c) partitions the new messages by severity and issues **one batched `INSERT … VALUES (…), (…)` per non-empty `log_*` table** (≤5). Regression-tested (`backend_test`: a re-finalize with no messages must delete the prior logs; `mark_done_routes_messages_to_severity_tables` finalizes a mixed-severity, mixed-*status* batch and asserts each message lands in its own table **and** each task receives its own status — covering the by-status grouping). **Remaining (separate, not blocking):** the reinsert is still unconditional (no diff/upsert to skip genuinely-unchanged logs) — a smaller optional win tracked under Arm 14 #3. |
| D-10 | S2 | 🟢 | **RESOLVED (2026-06-14) — dispatch recorded *after* the payload send → single-task loss under concurrency.** The ventilator popped a task from its queue, **streamed the payload to the worker, and only then** (`push_progress_task`, at the bottom of the loop) recorded the lease in the in-flight `progress_queue`. With a fast worker (echo, or any sub-millisecond conversion) the result could reach the **sink before that record existed**: the sink's `pop_progress_task(taskid)` missed it → took the "no such task id" branch and **discarded the result**, leaving the task stranded `Queued` (leased, never finalized) until the ≥1 h visibility-timeout reaper — i.e. effectively lost for the run (violates "no silent loss"). The push completes (mutex released) before the first content frame is sent, so the worker cannot return a result before the task is tracked — the race is eliminated, not narrowed. Recording-first also leaves a mid-stream send failure correctly in-flight for the reaper (the prior send-fail path relied on the startup `clear_limbo` reset instead). **Verified:** 18 consecutive clean bench runs at the previously-failing concurrencies (12×8-worker + 6×16-worker, 20000 tasks each, 0 loss). | D-9 | S2 | 🟢 | **RESOLVED (2026-06-14) — unsupervised sink/finalize threads → silent stall.** The manager (`dispatcher/manager.rs`) restart-loops only the **ventilator**; the **sink** and **finalize** threads are spawned **once**.
